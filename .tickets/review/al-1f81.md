# Workspace: refactor structure + clarify value proposition

The workspace subsystem is one of Loom's most differentiating surfaces, but the current Python layout and internal layering make it harder than it needs to be to extend safely.

This epic covers two tracks:
1) Refactor: reorganize workspace code + extract cross-cutting utilities into `agent_loom.core`.
2) Product: tighten the workspace value proposition beyond 'just use git', focusing on annotations, sandboxes, TTLs, lifecycles, cleanup/GC, and the polyrepo harness (control plane) model.

## Design

Constraints / invariants to preserve:
- Two orthogonal operating models: repo mode (`loom workspace ...`) and poly control-plane mode (`loom workspace poly ...`). No implicit dispatch.
- CLI output is a contract (tests under `tests/test_workspace_*.py`).
- Prefer deterministic ordering and explicit intent for multi-repo operations.

Primary outcomes:
- Workspace package organization reads like a product (repo vs poly/harness) and a layered system (git plumbing vs workspace domain).
- Common helpers (exec, io, time/duration, fs-safe encoding, concurrency) live in `agent_loom.core` and are reused across Loom subsystems.
- Workspace lifecycle and metadata become first-class: unified schema, clear storage locations, and a coherent cleanup/GC story.
- Documentation/cookbook leads with the value proposition and user stories, not just command enumeration.

## Acceptance Criteria

- A ticket set exists (children of this epic) that fully covers the refactor and the value-prop work.
- Each child ticket has: scope, non-goals, step-by-step plan, acceptance criteria, verification commands.
