# Workspace: reorganize package layout + split CLI

`src/agent_loom/workspace/` is feature-rich but hard to navigate: poly-only code sits alongside repo-mode code; multiple `*_ops.py` modules overlap; and `cli.py` has grown into a large monolith.

This ticket is a pure re-organization/refactor: no behavior changes intended.

## Design

Proposed structure (example; adjust to fit reality during implementation):

- `src/agent_loom/workspace/`
  - `cli.py` becomes thin entrypoint that wires subcommands.
  - `repo/` repo-mode domain operations
    - `worktrees.py` (add/ensure/rm/ls/status/check-clean/divergence)
    - `snapshots.py`
    - `cleanup.py`
    - `sandboxes.py`
    - `init.py`
  - `poly/` (a.k.a. harness control plane)
    - `init.py`
    - `inventory.py` (workspace.json repo CRUD + sync)
    - `worktrees.py` (group worktree add/rm/ls/status/diff/rebase/push/prune)
    - `snapshots.py`
    - `services.py` + `deps.py`
    - `leases.py` + `gc.py`
    - `sandbox.py`
  - `git/` workspace-scoped git plumbing (parsing porcelain, safe guards)
  - shared: `models.py`, `errors.py`, `constants.py`, `guards.py`, `state.py` (workspace.json), `meta.py` (annotations)

Implementation steps:
- Move functions file-by-file without changing signatures; update imports.
- Keep the public import surface stable via `src/agent_loom/workspace/core.py` (re-export) during the refactor.
- Split `src/agent_loom/workspace/cli.py` into small modules under `workspace/cli/` or `workspace/commands/` and keep the CLI UX identical.
- Update tests only when the contract intentionally changes (not expected in this ticket).

Non-goals:
- No new features.
- No storage layout changes.
- No CLI renames.

## Acceptance Criteria

- Workspace code is split by mode and concern (repo vs poly/harness; domain vs git plumbing).
- `src/agent_loom/workspace/cli.py` is <= ~250 lines and delegates to submodules.
- Gates pass: `uv run basedpyright`, `uv run ruff check .`, `uv run pytest tests/test_workspace_*.py`.
