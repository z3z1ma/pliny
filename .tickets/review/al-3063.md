# Workspace: change/impact analysis across repos (snapshots + service deps)

To justify workspace poly/harness beyond 'run git in a loop', we should make cross-repo context *actionable*.

We already have:
- services metadata (`services/<repo>.md`) + derived deps index (`services/index.json`)
- poly snapshots (capture/diff/restore)
- poly exec/diff

This ticket turns those into a simple impact-analysis workflow.

## Design

Add a harness-level command that answers:
- "Given a set of changes, what services are impacted?"

Possible approaches:
1) Snapshot-based:
- `loom workspace poly snapshot diff <name>` already knows which repos changed (via commit/dirty state).
- Map changed repos -> impacted via deps graph (`used_by_transitive`).
- Output: changed repos + impacted repos + suggested repo sets.

2) Diff-based (optional):
- For a group worktree, use `worktree diff` results to identify which repos have changes.
- Same mapping.

UX shape:
- `loom workspace poly impact <snapshot|group|repos...>` (name TBD)
- JSON output for automation + a human-readable table.

Non-goals:
- No file-level dependency scanning; rely on the curated services metadata.

## Acceptance Criteria

- A new command (or an extension to existing deps/snapshot commands) produces a deterministic impacted-services report.
- Includes tests with a small fake workspace verifying: changed={a} implies impacted includes reverse deps.
