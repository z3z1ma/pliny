# Mission

Create the first `loom-drive` product surface for Loom's objective-driven parent
loop.

This is a protocol-authority iteration. The goal is not to implement a daemon,
script, CLI, scheduler, or Gastown clone. The goal is to add a Markdown-native
workflow-coordinator skill that teaches the current chat agent how to drive a
high-level objective through Loom owner records, Ralph/subagents, evidence,
critique, and continuation without requiring the user to manually prompt every
step.

# Source Snapshot

Key constraints from `constitution:main`:

- Loom is not a runtime, daemon, model router, dashboard, MCP bundle, or product
  CLI.
- Tickets remain the sole execution ledger.
- Packets are bounded execution contracts, not transcript dumps.
- Parent workflow judgment and child bounded execution should remain explicitly
  separate.
- Scope and write authority must fail closed rather than being guessed.
- New workflows should first be expressed as disciplined routes through existing
  owner layers, not as new canonical layers.

Key desired behavior from `spec:objective-driven-parent-loop`:

- A high-level chat request activates an objective-driven parent-loop workflow.
- The agent clarifies enough with the user to establish measurable
  initiative-level objectives, then proceeds autonomously through existing Loom
  owners.
- The parent repeatedly creates/refines durable records, decomposes the next
  tranche into bounded tickets, executes via local edits or Ralph/subagents,
  reconciles ticket/evidence/critique/wiki truth, and decides whether to stop,
  ask the user, or create the next tranche.
- A dedicated outer-loop subagent is allowed for fresh synthesis, but it does not
  own objective truth or execution state.
- No script, daemon, outside poller, product CLI, dashboard, hidden database, or
  new canonical layer may be introduced.

Acceptance targets from `ticket:odpl001`:

- tells agents when to activate `loom-drive`
- tells agents how to ask enough focused user questions to establish measurable
  objective/success criteria without requiring approval for every downstream step
- routes durable truth to existing owner layers
- defines the repeated loop: shape objective, plan tranche, create tickets,
  execute bounded work, reconcile, reassess objective, continue or stop
- defines optional dedicated outer-loop subagent use and requires parent
  reconciliation of its output
- preserves no-script/no-daemon/no-new-layer
- adds existing skill routing links where appropriate
