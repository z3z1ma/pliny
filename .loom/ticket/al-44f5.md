# Implement OMP model preflight and fail-fast sidecar behavior

Prevent opaque worker crash loops when an unavailable OMP model is configured.

## Acceptance Criteria

Invalid OMP models produce actionable failure state without repeated respawn churn, and valid models continue to launch.

## Notes

**2026-02-15T19:35:45Z**

Execution brief:

Objective alignment
- This is the direct fix path for al-7d1d: worker panes must fail with a clear actionable reason when model config is invalid, instead of entering respawn loops that stall sprint execution.

Scope
- Add OMP model preflight/validation in Team sidecar launch path.
- Convert invalid-model startup into deterministic fail-fast behavior (no repeated respawn).
- Emit actionable diagnostics (event + inbox message + log) that identify model + recipient + remediation command.
- Preserve normal launch behavior for valid models.

Non-goals
- No changes to non-OMP harness launch behavior.
- No redesign of broader retry policy for unrelated rc!=1 failures.
- No ticket/workspace architecture refactor.

Implementation plan
1) Add helper(s) in src/agent_loom/team/core.py:
   - `_omp_model_available(bin: str, model: str) -> tuple[bool, str]` using `subprocess.run([bin, "--list-models", model], ...)`.
   - Parse stdout/stderr robustly; treat "No models matching" as unavailable.
2) Integrate preflight in `tui(...)` OMP branch before spawning child:
   - If model empty -> return explicit TeamError path with clear message.
   - If model unavailable -> write sidecar event (new event type `sidecar.omp_model_invalid`), append sidecar log line, notify manager inbox once, and exit with `exit_reason="model_invalid"` without respawn loop.
3) Keep safe fallback behavior:
   - If preflight command itself fails due environment/provider issues (not "model missing"), emit warning event and continue to spawn once so we do not block valid private setups.
4) Add regression tests:
   - Extend tests/test_team_harness_omp.py to cover invalid-model and valid-model branches with mocked subprocess responses.
   - Add assertion that invalid model does not trigger repeated respawn path.
5) Add/update docs:
   - src/agent_loom/team/README.md troubleshooting section for OMP model validation/remediation.

Acceptance criteria
- Configured invalid OMP model yields clear model-invalid diagnostics and stops sidecar churn.
- Manager receives actionable remediation guidance including model and command path.
- Valid OMP model launch path remains functional (existing tests still pass).
- New tests fail without fix and pass with fix.

Risks / edge cases
- Some providers may not list all models via `--list-models`; detect provider/query failures separately from definite "not found" and avoid false-negative blocking.
- Repeated notifications can spam manager; ensure message is sent once per invalid-start cycle.
