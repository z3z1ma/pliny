# Pack built-in coverage expansion + operator docs refresh

## Objective alignment
Improve confidence and maintainability of the pack module by broadening test coverage across built-in packs and aligning docs with real CLI behavior.

## Scope
- Expand tests to cover `loom-compound-core` and `loom-team-core` lifecycle/invariant behavior.
- Add built-in pack contract checks (manifest validity + expected critical files).
- Refresh pack docs to reflect current commands, drift diff workflow, and lockfile expectations.

## Non-goals
- Authoring new built-in packs.
- Broad docs rewrite outside pack-focused surfaces.
- New CLI features unrelated to pack lifecycle/documentation.

## Implementation plan
1. Extend/add tests near:
   - `tests/test_pack_lifecycle.py`
   - `tests/test_pack_loom_agile_core_invariants.py` (or split into per-pack invariant modules).
2. Add assertions for each built-in pack:
   - list/load succeeds,
   - representative files exist,
   - install/uninstall smoke path works in temp repo.
3. Update documentation surfaces:
   - `src/agent_loom/pack/README.md`
   - any relevant top-level docs referencing `loom pack` behavior.
4. Ensure docs include explicit lockfile commit expectation and drift review workflow.
5. Keep docs/tests synchronized with actual CLI output contract.

## Verification commands
- `uv run ruff check .`
- `uv run basedpyright`
- `uv run pytest tests/test_pack_lifecycle.py`
- `uv run pytest tests/test_pack_loom_agile_core_invariants.py`
- `uv run pytest tests/test_pack_cli_ux.py`

## Acceptance criteria
- Built-in pack coverage no longer relies on only sample/agile paths.
- Pack docs match current command behavior and flags.
- Regression suite catches pack manifest/content drift for built-in packs.

## Risks / edge cases
- Invariant tests can become brittle if pack content intentionally changes.
- Mitigation: assert contract-level invariants (presence/patterns), not incidental wording.
