# Team core: extract worker/profile planning primitives

Objective alignment:
Spawn/resume flows depend on headcount limits, role profile selection, and workspace assignment helpers that are currently dispersed in `team/core.py`. Consolidating these primitives improves modularity and reduces risk when adding roster/runtime features.

Scope:
- Introduce a worker planning domain module (for example `src/agent_loom/team/worker_planning.py`) that owns:
  - headcount calculations (`max_headcount`, active spawn counts),
  - role agent/model resolution helpers,
  - always-on profile fallback/workspace mapping helpers used by spawn paths.
- Update `spawn`, `resume_worker`, `spawn_integrator`, and `spawn_persona` orchestration in `team/core.py` to delegate to the planning module where appropriate.
- Preserve existing lifecycle enforcement and run-state schema compatibility.
- Update tests in `tests/test_team_spawn_headcount.py`, `tests/test_team_spawn_yaml_members.py`, and `tests/test_team_spawn_integrator.py`.

Non-goals:
- No new lifecycle rules or role semantics.
- No changes to tmux pane/window creation behavior outside delegation boundaries.
- No CLI argument changes.

Implementation plan:
1) Identify planning helpers currently embedded in core and define stable contracts for the new module.
2) Move pure planning logic first; keep side-effecting spawn orchestration in core.
3) Replace direct helper usage in spawn-family flows with module calls, minimizing behavior deltas.
4) Expand tests around headcount limits, duplicate-id handling, role profile resolution, and workspace assignment.
5) Remove dead helper definitions/imports from core.

- uv run basedpyright

Risks/edge cases:
- Off-by-one headcount regressions can permit oversubscription.
- Built-in profile fallback logic can drift from roster constraints if contracts are unclear.
- Workspace mapping errors can route workers to wrong worktrees; tests must assert resolved paths explicitly.

## Acceptance Criteria

- Worker/profile planning logic is centralized in a dedicated module and invoked by spawn-family orchestration.
- Spawn/resume/integrator behavior remains stable under existing regression tests.
- No unused planning helpers remain in `team/core.py`.
