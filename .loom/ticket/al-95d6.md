# TEAM sprint auto-tag not propagated to spawned worker ticket contexts

Objective alignment:
Sprint execution depends on deterministic ticket tagging. Missing sprint auto-tag forces manual correction and breaks cheap-worker autonomy.

Current finding (verified):
- `loom ticket create` auto-tagging relies on ticket sprint context first, then `TEAM_SPRINT_TAG` env (`src/agent_loom/ticket/core.py`, `_effective_sprint_tag`).
- Team sets `TEAM_SPRINT_TAG` only when a tmux pane is created. Existing always-on panes (architect/manager) keep stale env if sprint is set later.
- `prep_sprint`/`sprint_set` update run sprint state, but do not synchronize ticket sprint context in `.loom/ticket/config.yaml`.
- Result: ticket creation from pre-existing panes after sprint changes can miss sprint tag unless manually provided.

Scope:
- Ensure sprint changes in Team (`prep_sprint`, `sprint_set`, `sprint_clear`) synchronize ticket sprint context in the canonical run ticket dir.
- Add regression tests proving ticket creation in stale-pane-like environments still applies current sprint tag through synchronized ticket context.
- Keep command UX unchanged (no new flags/commands).

Non-goals:
- No tmux pane respawn strategy to refresh env vars.
- No cross-subsystem coupling where ticket module reads Team run.json directly.

Implementation plan:
1) Add a small Team helper to persist/clear ticket sprint context in `tickets_dir/config.yaml` using ticket APIs/core primitives.
2) Call helper from `prep_sprint`, `sprint_set`, and `sprint_clear` immediately after run sprint mutation succeeds.
3) Preserve existing behavior where explicit `--no-sprint-tag` still opts out in ticket create.
4) Add tests:
   - Team sprint command tests assert ticket sprint context sync is invoked with expected name/tag.
   - Integration-style test: simulate stale `TEAM_SPRINT_TAG` env and verify `loom ticket create` still gets sprint tag after Team sprint set/prep.
5) Verify no behavior regressions in existing sprint/team flows.

Verification:

- uv run ruff check .

Risks/edge cases:
- Multiple concurrent sprint mutations could race on `config.yaml`; use existing ticket store locking path or atomic write semantics.
- Clearing sprint must remove prior context cleanly (avoid stale tags after `sprint_clear`).
- Keep ticket-dir resolution canonical (repo-root `.loom/ticket`) to avoid writing context in worker-local paths.

## Acceptance Criteria

- Team sprint mutations (`prep_sprint`, `sprint_set`, `sprint_clear`) keep ticket sprint context synchronized in canonical ticket config.
- Ticket creation from panes with stale/empty `TEAM_SPRINT_TAG` still receives current sprint tag unless `--no-sprint-tag` is used.
- Regression tests cover this path and pass.
