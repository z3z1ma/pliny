# 10x Autoresearch Program

This file is the human-owned autoresearch program. Autoresearch agents read it
before running experiments. They do not edit this file during research unless a
human explicitly asks for a program change.

The loop controller is the LLM reasoning engine. Python utilities only run one
registered trial, validate contracts, render trial artifact reports, or run
diagnostics.

## Goal

Improve and prove `SKILL.md` by repeatedly testing candidate instruction changes
against clean-room subject-agent trials.

Each experiment has two jobs:

- test a specific hypothesis about current or candidate 10x behavior;
- add to the reusable eval/regression suite that shows the skill is still
  working.

The objective is not to maximize one hidden aggregate. Improve the rubric
criteria while preserving the discipline 10x exists to enforce:

- outer-loop ambiguity handling;
- durable record creation;
- ticket decomposition;
- evidence integrity;
- minimal implementation;
- closure coherence;
- retrospective learning;
- research method discipline.

Simplicity matters. A neutral result that deletes complexity can be worth
keeping as a simplification win.

## Editable Surfaces

You may edit experimental candidate artifacts, such as:

- `autoresearch/candidates/<candidate-name>.md`;
- temporary candidate copies of `SKILL.md` under an experiment branch or output
  directory;
- experiment records under `.10x/research/` when defining the next run;
- evidence and review records that preserve trial interpretation.

You must not edit during an autonomous research run:

- `autoresearch/program.md`;
- canonical `SKILL.md`;
- score or scenario catalogs;
- validators or harness code;
- accepted decisions or active specifications.

Canonical `SKILL.md` promotion requires explicit human approval after evidence
and review. Candidate artifacts are allowed to be messy while exploring.

## Setup

Before starting a research run:

1. Choose a run tag, for example `2026-06-27-s006-closure`.
2. Create or identify a dedicated working branch for the run.
3. Read `SKILL.md`, this file, `autoresearch/README.md`, and the relevant
   rubric, scenario catalog, and `autoresearch/trial-seeds/index.json` entries.
4. Confirm canonical files are clean after setup is committed:

```bash
python3 autoresearch/canonical_guard.py --require-clean
```

5. Establish the baseline by running current canonical instructions or the
   current accepted candidate before trying mutations.

Do not begin a multiday run while `SKILL.md` or `autoresearch/program.md` are
dirty in git. The run can approach 99% confidence on canonical-file safety only
when setup is committed and every experiment uses `--require-clean-canonical`.

## Single Iteration

One autoresearch iteration is:

1. State the hypothesis in a sentence.
2. Create or edit exactly one candidate artifact, or state that this is a
   current-skill regression trial with no candidate mutation.
3. Select the seed from `autoresearch/trial-seeds/index.json`; read its material
   records and material source files before writing the prompt.
4. Register or update exactly one experiment definition, including its
   scientific contract: question, hypothesis, expected behavior, inspection
   criteria, quality floor, and verdict record path.
5. Run one experiment with `autoresearch/run_once.py`.
6. Read `summary.json`, `plan.json`, `report.md`, raw trial artifacts, command
   artifacts, workspace manifests, and changed files.
7. Inspect the result against the registered scientific contract, rubric, and
   scenario expectations.
8. Record the verdict, limits, and evidence references in durable `.10x/`
   research, evidence, or review records.
9. Decide whether to keep, discard, mutate, branch, retry, or promote for human
   review.

The command shape is:

```bash
python3 autoresearch/run_once.py --experiment path/to/experiment.json --out .10x/evidence/.storage/<run-tag>/<experiment-id> --require-clean-canonical
```

For MICRO experiments, use scenarios that isolate one behavior or failure mode.
MICRO is for fast, focused optimization. MICRO still executes the subject
harness; it is narrow because the scenario is narrow.

For FULL experiments, use broader scenario coverage in the subject harness.
FULL is for checking whether a candidate that looked good in focused tests still
behaves coherently across the broader surface.

`run_once.py` writes `<out>/canonical_guard.json`. Before considering a result,
confirm that `unchanged_during_run` is true. If the guard fails, discard the
iteration and restore canonical files before continuing.

## Handling Subject Questions

Subject agents may ask clarifying questions. That is normal evidence, not a
failed run. The LLM researcher decides the next turn after reading the raw
transcript; Python does not decide whether the conversation is complete.

When a subject asks a material question:

1. Read the raw transcript for each arm.
2. Decide whether the answer is available from the scenario intent, existing
   records, or a reasonable benchmark assumption.
3. If the answer is available, register a continuation experiment with
   `prior_raw_paths` pointing to each arm's prior raw artifact.
4. Provide the next user message through `prompts_by_arm` when arms asked
   different questions or need different wording. Use a shared `prompt` only
   when the same reply is correct for every arm.
5. Run exactly one continuation turn with `run_once.py`.
6. Inspect the combined transcript and decide whether to continue, mark
   inconclusive, or ask the human for a real product decision.

Do not prewrite fixed arrays of follow-up inputs. LLMs may ask the same
questions in different orders or ask different questions entirely. Continuation
is an autoresearch reasoning step, not runner control flow.

## Reading Trial Artifacts

After each run, inspect:

- `<out>/summary.json`;
- `<out>/plan.json`;
- `<out>/report.md`;
- raw artifacts under `<out>/raw/`;
- command metadata, stdout JSONL, stderr, and last messages under
  `<out>/codex/`;
- prompts under `<out>/prompts/`;
- workspace manifests and archived workspaces under `<out>/workspaces/`;
- `<out>/canonical_guard.json`.

Confirm that the `scientific_contract` in the plan, summary, raw artifacts, and
report matches the question you meant to run. The report is only a secondary
view. The scientist's judgment lives in durable `.10x/` records with direct
artifact references.

Use statuses such as:

- `keep`: rubric movement and complexity tradeoff justify keeping the candidate;
- `discard`: behavior is worse, null, confounded, or too complex;
- `mutate`: result suggests a follow-up candidate rather than direct keep;
- `crash`: the run failed before producing usable evidence;
- `review`: the result is promising but needs human or adversarial review;
- `regression-pass`: current or candidate behavior still satisfies the rubric.

## Keep Or Discard

Keep a candidate only when:

- it improves the targeted rubric criterion or preserves behavior while
  simplifying;
- it does not trigger quality floors;
- the report and raw artifacts support the claim;
- the complexity cost is acceptable;
- promotion boundaries are respected.

Discard or revert when:

- a targeted behavior regresses;
- a floor trigger makes the candidate non-promotable;
- the result is null or confounded;
- the candidate improves the measurement story by exploiting the setup rather
  than improving actual 10x behavior;
- the implementation makes `SKILL.md` harder to understand without a meaningful
  behavioral gain.

If a candidate crashes because of a simple mechanical error, fix the candidate
and rerun once. If the idea itself is broken, record `crash` and move on.

## Scenario Splits

Use `autoresearch/splits/skill-improvement-v1.json`.

- Use `exploration_scenarios` for candidate search and MICRO optimization.
- Do not tune directly against `held_out_scenarios`.
- Before marking a candidate `review`, run the review scenario set or explain
  why a narrower review is sufficient.
- A candidate that improves exploration behavior but regresses held-out
  scenarios is not promotion-ready.

Held-out scenarios increase confidence that the candidate improves 10x behavior
rather than exploiting one seed state or rubric weakness. They do not make
automated promotion proof 99%.

## Infinite Loop

Once setup is complete, continue autonomously until the human interrupts:

1. inspect the current best candidate and recent durable records;
2. choose the next hypothesis;
3. create or mutate one candidate;
4. run one MICRO or FULL experiment;
5. inspect trial artifacts and record a verdict;
6. keep, discard, mutate, branch, or retry;
7. repeat.

Do not ask whether to continue after every experiment. The purpose of
autoresearch is that the LLM reasoning engine keeps thinking and experimenting.

If you run out of ideas:

- inspect the weakest rubric criterion;
- read the scenario catalog entries and recent trial records for that behavior;
- compare near-miss candidates;
- simplify the candidate;
- test a narrower MICRO;
- combine two previously useful ideas;
- try a more radical candidate in a separate branch.

Before asking for human review, run at least these checks:

- baseline comparison;
- targeted MICRO;
- one ablation with the candidate's most suspicious paragraph removed;
- one simpler variant;
- held-out or review scenario check;
- `git diff -- SKILL.md autoresearch/program.md` must be empty.

## Boundaries

Every claim must be tied to raw artifacts, command/workspace artifacts, or
recorded evidence.

Do not promote canonical `SKILL.md` automatically. The autonomous loop produces
candidate evidence. The human decides promotion.

The safety process can approach 99% confidence that canonical files are not
accidentally modified. It cannot honestly make discovery or automated
promotion-proof 99%; those still require stronger live evaluations, larger
review sets, adversarial review, and human promotion.
